# 体系的デバッグスキル (Systematic Debugging)

Based on [obra/superpowers](https://github.com/obra/superpowers/blob/main/skills/systematic-debugging/SKILL.md)

## Overview

ランダムな修正は時間を浪費し、新たなバグを生む。表面的なパッチは根本問題を隠す。

**核心原則:** 修正を試みる前に**必ず根本原因を特定する**。症状の修正は失敗である。

> ⚠️ Dockerコンテナレベルの問題は `docker-debug` スキルを使用すること。

## 鉄の掟

```
根本原因調査なしの修正は禁止
```

Phase 1 を完了していなければ、修正を提案してはならない。

## いつ使うか

あらゆる技術的問題に使用:
- テスト失敗
- 本番バグ
- 予期しない挙動
- パフォーマンス問題
- ビルドエラー
- インテグレーション問題

**特にこういう時こそ使え:**
- 時間的プレッシャーがある時 (焦りが推測を誘う)
- 「ちょっとした修正」が明白に見える時
- すでに複数の修正を試した後
- 前回の修正がうまくいかなかった時
- 問題を完全に理解していない時

## 4つのフェーズ

各フェーズを順番に完了すること。

### Phase 1: 根本原因調査

**修正を試みる前に:**

1. **エラーメッセージを注意深く読む**
   - エラーや警告を読み飛ばさない
   - スタックトレースを最後まで読む
   - 行番号、ファイルパス、エラーコードを記録

2. **確実に再現する**
   - 正確な再現手順は？
   - 毎回発生するか？
   - 再現できない場合 → データを集める、推測しない

3. **最近の変更を確認する**
// turbo
   ```powershell
   git diff HEAD~5 --stat
   git log -n 10 --oneline
   ```

4. **マルチコンポーネントシステムでの証拠収集**

   コンポーネント境界ごとに診断を挿入:
   ```
   各境界で:
   - 入力データをログ
   - 出力データをログ
   - 環境/設定の伝播を検証
   - 各レイヤーの状態を確認
   ```

   Docker環境の場合:
   ```powershell
   # コンテナログ確認
   ./scripts/dev_manager.ps1 logs <service>
   
   # コンテナ間疎通
   docker exec <container> curl -s http://<service>:<port>/health
   ```

5. **データフロー追跡**
   - 不正な値はどこで生まれた？
   - 誰がその値を渡した？
   - **ソースまで遡る → `root-cause-tracing` スキルへ**

### Phase 2: パターン分析

1. **動作する例を見つける**
   - 同じコードベース内で類似の動作コードを探す
   - 何が動いていて何が壊れているか？

2. **リファレンスと比較する**
   - パターンを実装する場合、リファレンス実装を**完全に**読む
   - 流し読みしない — 1行ずつ全部読む

3. **差分を特定する**
   - 動作するものと壊れているもので何が違うか？
   - すべての差異をリストアップ
   - 「それは関係ない」と決めつけない

### Phase 3: 仮説とテスト

**科学的方法:**

1. **仮説を1つ立てる**
   - 明確に述べる: 「Xが根本原因だと思う。なぜならY」
   - 書き留める — 具体的に

2. **最小限の変更でテスト**
   - 仮説を検証する最小の変更を行う
   - 一度に1つの変数のみ
   - 複数のことを同時に修正しない

3. **続行前に検証**
   - 成功 → Phase 4へ
   - 失敗 → **新しい仮説**を立てる
   - 追加の修正を重ねない

### Phase 4: 実装

1. **失敗するテストケースを作成**
   - 最もシンプルな再現
   - 自動テストが望ましい
   - 修正前に必ず作成

2. **単一の修正を実装**
   - 特定した根本原因に対処
   - 一度に1つの変更
   - 「ついでに」の改善はしない

3. **修正を検証**
// turbo
   ```powershell
   pnpm turbo run test --filter=<app>
   ```

4. **修正がうまくいかない場合**
   - **停止**
   - 試した修正の回数を数える
   - 3回未満: Phase 1 に戻り、新情報で再分析
   - **3回以上: 停止してアーキテクチャを疑う**
   - ユーザーに相談 (`notify_user`) してから次の修正を試みる

## レッドフラグ — 停止してプロセスに従え

以下の考えが浮かんだら**停止**:
- 「とりあえず修正して、後で調査」
- 「Xを変えて動くか見てみよう」
- 「複数の変更をまとめてテスト実行」
- 「テストをスキップして手動確認」
- 「多分Xが原因、直してみよう」
- 「完全に理解してないけどこれで動くかも」
- 「もう1回修正を試す」(すでに2回以上試した後)

**すべて → Phase 1 に戻る**

## よくある言い訳

| 言い訳 | 現実 |
|:---|:---|
| 「単純な問題、プロセス不要」 | 単純なバグにも根本原因がある |
| 「緊急、プロセスの時間がない」 | 体系的デバッグは推測&修正の繰り返しより**速い** |
| 「まずこれを試して、それから調査」 | 最初の修正がパターンを決める。最初から正しくやれ |
| 「修正の確認後にテスト書く」 | テストなしの修正は定着しない |
| 「複数修正を同時に行えば時間節約」 | 何が効いたか分離できない。新バグの原因になる |

## クイックリファレンス

| フェーズ | 主要活動 | 成功基準 |
|:---|:---|:---|
| **1. 根本原因** | エラー読む、再現、変更確認、証拠収集 | WHATとWHYを理解 |
| **2. パターン** | 動作例を見つけ、比較 | 差異を特定 |
| **3. 仮説** | 理論を立て、最小限テスト | 確認 or 新仮説 |
| **4. 実装** | テスト作成、修正、検証 | バグ解消、テストPASS |
