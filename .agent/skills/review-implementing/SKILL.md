# 実装レビュースキル (Review Implementing)

Based on [mhattingpete/claude-skills-marketplace](https://github.com/mhattingpete/claude-skills-marketplace/blob/main/engineering-workflow-plugin/skills/review-implementing/SKILL.md)

コードレビューのフィードバック、PR指摘、実装プランの仕様照合を体系的に処理する。

---

## いつ使うか

- ユーザーがレビューコメントやフィードバックを提示した時
- PRレビューの指摘事項を実装する時
- `implementation_plan.md` の自己レビューを行う時
- 「このコメントを対応して」「フィードバックを実装して」と言われた時

---

## 体系的ワークフロー

### Step 1: フィードバックの解析

レビュー内容を個別の項目に分解:

- 番号付きリストを分割 (1., 2., ...)
- 箇条書きや非番号フィードバックを処理
- 個別の変更要求を抽出
- **曖昧な項目は作業前にユーザーに確認**

### Step 2: タスクリストの作成

フィードバック項目をアクション可能なタスクに変換:

```markdown
| # | フィードバック | タスク | 状態 |
|:--|:---|:---|:---|
| 1 | 型ヒントを追加 | `extract()` に型アノテーション追加 | [ ] |
| 2 | 重複タグ検出を修正 | ロジックの条件式を修正 | [ ] |
| 3 | docstring更新 | chain.py のdocstringを更新 | [ ] |
| 4 | エッジケーステスト追加 | 空配列と null 入力のテスト作成 | [ ] |
```

- 各フィードバックを1つ以上のタスクに
- 複雑なフィードバックは小さなタスクに分割
- 具体的で測定可能なタスクにする

### Step 3: 変更の実装 (1つずつ)

各タスクについて:

**① 関連コードを特定:**
```powershell
# grep で関数/クラスを検索
grep_search "<keyword>" --path apps/<app>/src --includes "*.ts,*.tsx"
```

**② 変更を実施:**
- プロジェクト規約に従う (GEMINI.md)
- UIコンポーネント → `@repo/ui` から使用
- 設定 → `@repo/config` を継承
- 既存の機能を壊さない

**③ 変更を検証:**
// turbo
```powershell
pnpm turbo run test --filter=<app>
pnpm turbo run lint
```

**④ タスクを完了にマーク:**
- 完了後すぐにステータス更新
- **1つずつ** — 複数を同時に進行しない

### Step 4: フィードバックの種類別対応

| 種類 | 対応方法 |
|:---|:---|
| **コード変更** | 既存コードを編集。型ヒントは PEP 604/585 準拠 |
| **新機能要求** | 必要に応じて新ファイル作成。テストも追加 |
| **ドキュメント** | docstring更新、markdown修正 |
| **テスト追加** | pytest (Python) / vitest (TypeScript) を使用 |
| **リファクタリング** | 機能を保持しつつ構造改善。テストで回帰確認 |

### Step 5: バリデーション

すべての変更実装後:
// turbo
```powershell
# テスト実行
pnpm turbo run test --filter=<app>

# Lint チェック (TypeScript)
pnpm turbo run lint

# Lint チェック (Python)
pnpm lint:py
```

### Step 6: 報告

ユーザーに結果を報告:
- 変更リストの最終状態を共有
- 判断に迷った点を説明
- ブロッカーがあれば報告

## エッジケース

| 状況 | 対応 |
|:---|:---|
| **矛盾するフィードバック** | ユーザーに確認。矛盾を明確に説明 |
| **破壊的変更が必要** | 実装前にユーザーに通知。影響と代替案を議論 |
| **変更後にテスト失敗** | テスト修正完了前にタスクを完了にしない → `test-fixing` スキル |
| **参照コードが存在しない** | ユーザーに確認。理解を検証してから進行 |

## implementation_plan.md 自己レビュー

PRフィードバックの対応だけでなく、自分のプランもレビューする:

### GO/NO-GO ゲート (実装前チェック)

実装に着手する前に以下を確認。1つでも ❌ なら **NO-GO** → ユーザーに報告:

| # | チェック項目 | 判定基準 |
|:--|:---|:---|
| 1 | 技術的に実現可能か | 依存ライブラリ・API制約・環境の確認 |
| 2 | 既存機能への破壊的影響は許容範囲か | `impact-check` スキルで事前調査 |
| 3 | タスクは 50% 以下のコンテキストで完了可能か | 大きすぎる場合はサブタスクに分割 |
| 4 | 必要な情報はすべて揃っているか | 不明点はユーザーに確認 |

> NO-GO の場合: 理由・ブロッカーを明記し、代替案を提示すること。

### 完全性チェック
- [ ] 変更目的と背景が記載されている
- [ ] 対象ファイル一覧がある
- [ ] 各ファイルの変更内容が具体的
- [ ] 破壊的変更の有無が明記
- [ ] 検証計画がある
- [ ] **変更・削除する関数/シンボルの全呼び出し箇所を `grep_search` で網羅確認した**

### プロジェクト規約との照合
- [ ] `@repo/ui` からコンポーネントをインポート
- [ ] `@repo/config` から設定を継承
- [ ] `pnpm` を使用 (npm/yarn 禁止)
- [ ] Docker操作はスクリプト経由
- [ ] Python: Ruff ルール準拠

### リスク評価

| リスクレベル | 基準 | 対応 |
|:---|:---|:---|
| 🟢 低 | UIテキスト変更、スタイル調整 | そのまま実行 |
| 🟡 中 | ロジック変更、API変更 | テスト必須 |
| 🔴 高 | DBスキーマ変更、認証変更 | ユーザー確認必須 |

### よくある見落とし
- [ ] 型定義の更新 (TypeScript)
- [ ] エクスポートの追加 (index.ts barrel files)
- [ ] 環境変数の追加 (`.env.example` の更新)
- [ ] Docker設定の更新 (volumes, ports, env)
- [ ] ドキュメントの更新 (README, PROJECT_MAP, ADR)
- [ ] 同一関数の複数ページからの呼び出し (grep で全箇所確認)
- [ ] レンダリングパスからのバックグラウンド処理分離 (cron/スケジューラへの移行検討)

## 大規模機能プランニング

複数ファイル・UIフロー変更を伴う大きな機能開発時は、以下の3視点で分析する。
通常の `implementation_plan.md` 内にセクションとして記載すること。

### いつ使うか
- 変更ファイルが **5つ以上** になりそうな場合
- 新規画面/ページの追加を伴う場合
- DB スキーマ変更 + API + UI の3層に跨がる場合

### 3視点テンプレート

`implementation_plan.md` の `## Proposed Changes` の直前に以下を追加:

```markdown
## 要件分析 (PM視点)
- ユーザーストーリー: 〇〇として、△△したい。なぜなら□□だから。
- 受け入れ基準:
  - [ ] 基準1
  - [ ] 基準2
- スコープ外: ここでは対応しないもの

## UXフロー (UX視点)
- 画面遷移: A → B → C
- 主要インタラクション: ボタン押下で○○が表示
- エラー状態: ネットワークエラー時の表示

## 技術設計 (Eng視点)
- アーキテクチャ: どの層に変更が入るか
- データフロー: フロント → API → DB の流れ
- パフォーマンス考慮: キャッシュ、ページネーション等
```

### フェーズ分割ガイド

大機能は **3フェーズ以内** に分割:

| フェーズ | 内容 | 基準 |
|:---|:---|:---|
| Phase 1 | 基盤 (DB/API) | 単独でテスト可能な最小単位 |
| Phase 2 | UI統合 | Phase 1 の上に構築 |
| Phase 3 | 仕上げ (テスト・エッジケース) | 品質保証 |

> 各フェーズが独立してコミット・検証できること。
